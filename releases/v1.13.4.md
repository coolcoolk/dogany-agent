# Release notes -- v1.13.4 (2026-07-21)

Type: PATCH (1 user-visible live-instance fix; 1 memory-correctness fix; no
migration required, no schema change).

## Summary

Patch on v1.13.3. Two baseline infra fixes shipped in commit bec2fb7.

The primary fix (DGN-480) stops false watchdog-triggered bot restarts on
relocated instances. When an instance is moved to a new path the watchdog's
registered LaunchAgent plist retains the old baked paths; the heartbeat file
is never written to the old location, so the watchdog detects a perpetual
stall and force-restarts the live bot roughly every 10 minutes while sending
a "service restarted" notification each time. No real fault was present; the
restarts and spam were entirely watchdog housekeeping drift. `watchdog_setup.sh`
now rewrites the plist paths on each setup call to reflect the instance's
current location.

The secondary fix (DGN-479) closes a memory-correctness regression introduced
in v1.13.1: a framework consume clobbered the `autoMemoryEnabled: false`
setting that DGN-431 had previously established, silently re-enabling the CC
auto-memory shadow store and allowing it to drift from the Dogany memory
engine. The fix re-ships `autoMemoryEnabled: false` in the template and adds a
`PreToolUse` hook (`cc-memory-write-guard.py`) that denies any write attempt
into the CC shadow store or the engine `memories/` directory, making the
off-switch clobber-proof against future framework consumes.

## Fixed

### DGN-480: bridge watchdog LaunchAgent re-registration on instance move (user-visible)

`scripts/bridge/watchdog_setup.sh` registers a LaunchAgent plist that bakes
the instance's absolute paths (heartbeat directory, log path, restart script
path) at setup time. When an instance is moved to a new location after the
watchdog was first armed, the registered plist retains the old paths. The
heartbeat file is written at the new location, but the watchdog checks the
old path -- it sees no heartbeat file and interprets the absence as a stall.

Result: the watchdog issues a force-restart of the live bot approximately
every 10 minutes (the heartbeat window), accompanied by a "service restarted"
push notification, indefinitely. The bot continues running but is repeatedly
kicked by its own watchdog with no real fault present.

The fix adds a path-rewrite pass to `watchdog_setup.sh`: on each invocation
the script reads the currently registered plist and updates any baked paths
that do not match the instance's current location before re-loading the plist.
The setup script is idempotent; running it on a non-moved instance is a no-op.

Instance impact: any instance that has been physically moved (new home
directory, renamed parent path, etc.) after initial watchdog setup is affected.
Run `watchdog_setup.sh` once against this release to re-register the plist with
correct paths. No data loss; the bot requires no restart.

---

### DGN-479: CC auto-memory shadow store off-switch hardened + write-guard hook

The Claude Code auto-memory shadow store is a CC-native parallel write path
that persists conversation summaries into `.claude/projects/*/memory/`. Dogany
manages its own memory engine (`memories/`); the CC store is a divergent,
ungoverned parallel copy.

DGN-431 established `autoMemoryEnabled: false` in the instance `.claude/settings.json`
to disable the CC store. The v1.13.1 framework consume (`update.sh` skills /
settings refresh) clobbered this value, silently restoring `autoMemoryEnabled: true`.
From that point forward the CC store was active and diverging from the Dogany
memory engine without any agent-visible signal.

Two fixes ship in this release:

1. Template `settings.json` now carries `autoMemoryEnabled: false` in the
   framework baseline. Future `update.sh` runs source this from the template,
   so a consume can no longer restore the default-true value.

2. New `PreToolUse` hook `cc-memory-write-guard.py` (registered in
   `settings.json`): denies any write tool call whose target path is inside
   `.claude/projects/*/memory/` or `memories/`. Fail-open: if the hook itself
   errors, the write is permitted (agent continuity preserved). The hook
   provides a hard second layer that survives any future settings overwrite.

Instance impact: instances that consumed v1.13.1 or later and have not
manually restored `autoMemoryEnabled: false` may have an active CC shadow
store. After updating to v1.13.4 the store is disabled and the guard hook is
armed. Existing shadow-store content is not deleted automatically; owners who
want to purge the CC shadow entries may do so manually from `.claude/projects/`.
Memory engine data is unaffected.
