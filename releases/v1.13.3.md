# Release notes -- v1.13.3 (2026-07-21)

Type: PATCH (1 instance-affecting bugfix; 1 drift/normalization fix; 1 CLI
branding change; 1 brief UX adjustment; no migration required, no schema
change).

## Summary

Patch on v1.13.2. The primary instance-affecting fix closes a skills-bundle
token substitution gap introduced in `update.sh` step 3j (DGN-406): after
rsyncing the skills-bundle directory the mint-token substitution pass was
never applied, leaving dunder tokens (`__USER_LABEL__` etc.) raw in
non-pack bundle skills on any instance that had consumed a previous update.
A one-time `update.sh` run against this release heals affected instances.

Also ships: a relationship-care bundle key normalization that corrects a
3-way drift across 5 surfaces (DGN-468), the dogany CLI local-door branding
for terminal title and statusLine (DGN-276), and a morning-brief workout
card session label + schedule header title fix (DGN-382 + brief UX).

---

## Fixed

### DGN-406: update.sh 3j -- skills-bundle mint tokens now re-substituted on rsync (instance-affecting)

`scripts/update.sh` step 3j rsyncs the template `.claude/skills-bundle/`
directory to the instance. Prior to this fix the skills-bundle path was
absent from the section-4 find-and-substitute pass, so mint tokens
(`__USER_LABEL__`, `__AGENT_NAME__`, and others) were written raw into
non-pack bundle skills after every update -- a silent regression visible
only when an agent read the affected skill and received a dunder token
instead of the substituted value.

The fix adds `skills-bundle/` to the substitution target set, matching the
existing treatment of `.claude/skills/` and other refreshed directories.

Instance impact: live instances on v1.13.2 or earlier that have consumed
at least one `update.sh` run and have non-pack skills in their
`.claude/skills-bundle/` directory may have raw dunder tokens in those
skills. A single `update.sh` run against v1.13.3 applies the substitution
pass and resolves the condition. No manual file edits required.

Real-world evidence: unsubstituted `__USER_LABEL__` observed in live Warg
`workout-log/SKILL.md` (DGN-402 grill FATAL-2 evidence, 2026-07-18). The
root cause traces to the DGN-402 knowledge-wiring delivery which triggered
a skills-bundle refresh.

---

### DGN-468: relationship-care bundle key normalized (hyphen -> underscore, 5 surfaces)

A 3-way drift in the `relationship-care` bundle key was confirmed during
DGN-466 investigation:

1. `service/lifekit/bundle.conf` carried `BUNDLE_RELATIONSHIP_CARE=` (correct
   key, underscore) but `.template/config/lifekit.conf` had only
   `BUNDLE_RELATIONSHIP=` -- the `_CARE` suffix was absent, making the toggle
   unreachable on fresh instances.
2. i18n key used a hyphen form (`bundle.relationship-care`) inconsistent with
   all other bundle keys (underscore convention).
3. `agents/main/service/lifekit/bundle.conf` was missing the
   `relationship-care` row entirely (canonical-vs-live drift).

This fix normalizes all 5 surfaces to the underscore convention and closes
the missing-row gap in the main copy. Instances with the relationship-care
feature activated will see no behavior change; instances where the mismatch
silently suppressed activation can now enable it via the standard bundle
toggle.
