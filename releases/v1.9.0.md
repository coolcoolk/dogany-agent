# Release notes -- v1.9.0 (2026-07-17)

Type: MINOR (DGN-389 adds a user-facing feature; DGN-349/385/396 are fixes).

## Summary

Daily-retro health-source gate, update-check newer-than semantic fix,
update.sh reverse-drift protection for un-versioned components, and retro
Warg live-read freshness fix.

---

## Added

### DGN-389: daily-retro RETRO_HEALTH_SOURCE gate + task section

`routines/daily-retro.sh` gains two new capabilities:

- Config gate `RETRO_HEALTH_SOURCE=warg|local` (default `local` -- all
  existing instances are unaffected). When set to `warg`, the retro quotes
  the Warg agent's `report.section.retro` verbatim instead of computing
  diet/workout data locally. If the section has not arrived, a single-line
  fallback is shown ("워그 건강 리포트 미도착"). Instance Ag sets this to
  `warg` to comply with the 2026-07-14 health-domain transfer principle.

- Task section: done-today (`task-done-between`) and overdue
  (`task-overdue`) tasks are shown at the bottom of each retro. Overdue
  tasks are compressed to a few items + "외 N건" when the list is long.
  The section is silently omitted when there is no task data.

Template-canonical: both gates are comment-only in the template
`agent.conf`; instance values stay per-instance.

Prior to this release, Ag's daily-retro still computed diet/workout
locally (incl. a -1447 kcal line on dry-run) and had no task section
despite the task CLI (`task-done-between`, `task-overdue`) having shipped
in v1.3.0.

## Fixed

### DGN-385: update.sh reverse-drift protection for un-versioned components

`update.sh` gains two complementary guards for components that do not
carry an in-file version marker (such as `bridge/`):

1. Section-root hold: a path listed in `.claude/.dogany-preserve` that
   matches a top-level component root (e.g. `bridge/`) causes the entire
   rsync section for that component to be skipped, with a loud WARN
   showing the number of differing files. Previously, a root-level
   preserve entry generated zero rsync excludes and silently passed
   through -- the component was overwritten despite appearing in the
   preserve list.

2. Pin-based auto-detection: before refreshing `bridge/`, `update.sh`
   compares the instance `bridge/UPSTREAM.md` pin SHA against the
   template pin SHA. If the pins match but the local files differ from
   the vendor copy, the instance has local-ahead changes -- the section
   is skipped with a WARN ("preserve 등록 또는 업스트림 승격 힌트").
   If the pins differ, a re-vendor has occurred and the normal refresh
   proceeds (existing preserve entries are still respected).

Instances without a pin file or without `.dogany-preserve` entries are
unaffected -- behavior is byte-identical to previous releases.

Root cause: Metal's 2026-07-17 self-update (v1.5.3 -> v1.8.0) silently
overwrote `bridge/bot.py` and `bridge/formatting.py` (the DGN-372 fix),
because `bridge/` has no in-file version marker and was not covered by
the GUARDED_FILES mechanism. `database/lifekit.py` would have been
protected (its guard caught the same pattern in v1.4.0 / DGN-249);
`bridge/` had no equivalent. Immediate recovery: `git checkout --
bridge/` (no data loss, the fix was committed).

Design: V1 (UPDATE_HOLD_PATHS key in .instance.conf) was REDESIGN-
grilled (FATAL-2: policy file lost on re-mint; .dogany-preserve already
machine-recognized). LOCKED V2 uses the existing preserve manifest for
the root-hold and adds the pin-based auto-detection layer.

### DGN-349: update-check nudge gated on newer-than, not plain !=

`routines/version-check.py` now uses strict semantic newer-than
comparison (`_version_tuple` + `_is_newer`) for both the local check
and the remote check. Previously, the comparator used plain `!=`, which
fired when the public repo lagged the framework release (e.g. "built
from 1.8.0, upstream now has 1.7.1"). The nudge now fires only when the
other side is genuinely ahead. Equal versions and local-newer-than cases
are silent.

Verified: 12-case unit suite (lag, update, equal, numeric 1.10>1.9,
zero-pad, pre-release, unparseable) + E2E hook simulation (local 1.8.0
vs remote 1.7.1 -> SILENT; vs 1.9.0 -> FIRED).

Note: the public repo lagging the framework release by one minor is
expected behavior (public release is a separate step from the framework
commit). The fix handles this correctly without treating it as a defect.

### DGN-396: retro Warg mode uses live read via owner lifekit.sh, snapshot freshness fallback

When `RETRO_HEALTH_SOURCE=warg` and the retro fires, the routine now
calls Warg's official owner-side query script (`lifekit.sh`) at fire
time to get today's health aggregates live, instead of reading a
pre-generated snapshot file. This closes the cross-agent freshness gap
observed on 2026-07-17: Warg's snapshot was generated at 19:54, the
retro fired at 21:00, and meals logged in the intervening hour (1665
kcal shown vs 2184 actual) were missing from the retro.

On any failure (script missing, non-zero exit, empty/invalid output,
bounded timeout), the routine falls back to the existing snapshot file
and annotates the health section with its generation time ("HH:MM
기준"). The retro never fails or blocks on the cross-agent read.

Implementation: rides the DGN-389 surface (`RETRO_HEALTH_SOURCE` gate).
The relay round-trip (request-response handoff) pattern was explicitly
rejected for this mechanical read case; reserved for judgment cases.
Cross-agent data access case taxonomy documented in
`docs/CROSS-AGENT-ACCESS.md` (5-lane table + freshness-contract rule).
