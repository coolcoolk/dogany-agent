# Release notes -- v1.13.2 (2026-07-20)

Type: PATCH (1 bridge vendor fix; configurable SDK transport buffer size;
no migration required, no behavior change for users not hitting the 1MB limit).

## Summary

Same-day patch on v1.13.1. The SDK transport `max_buffer_size` knob was never
passed by the bridge, leaving all deployments at the 1MB SDK hard default.
A CLI->SDK JSON message exceeding 1MB (most commonly a base64 image inline in
a tool result) crashed the reader loop and silently lost the in-flight turn.
The fix adds an env-configurable default of 16MB (`CLAUDE_MAX_BUFFER_SIZE`;
existing deployments require no config change). Vendored upstream-first from
the OSS bridge repo (DGN-460).

---

## Fixed

### DGN-460: bridge -- SDK transport max_buffer_size now configurable (default 16MB)

`agents/.template/bridge/config.py` gains a new raw-env knob:

```
CLAUDE_MAX_BUFFER_SIZE = int(os.getenv("CLAUDE_MAX_BUFFER_SIZE", str(16*1024*1024)) or str(16*1024*1024))
```

`agents/.template/bridge/sdk_bridge.py`: `opts["max_buffer_size"] = CLAUDE_MAX_BUFFER_SIZE`
added to the `opts` dict in `_create_user_stream`, passing the value through
to `ClaudeAgentOptions`.

`agents/.template/bridge/.env.example`: `CLAUDE_MAX_BUFFER_SIZE=16777216`
documented under the session/timeout knobs block.

`agents/.template/bridge/UPSTREAM.md`: `Vendor-rev:` line bumped (DGN-460,
OSS push pending) per DGN-385 pin discipline -- so that `update.sh` correctly
detects the bridge content change and does not silently skip the rsync on
instances with a matching pin.

Background: `ClaudeAgentOptions.max_buffer_size: int | None` (SDK types.py)
has always existed; the bridge simply never passed it, leaving the SDK's
`_DEFAULT_MAX_BUFFER_SIZE = 1024*1024` in effect. The crash manifests as:

```
claude_agent_sdk._internal.query - ERROR - Fatal error in message reader:
Failed to decode JSON: JSON message exceeded maximum buffer size of 1048576 bytes...
bridge.sdk_bridge - ERROR - Reader loop crashed for user <workspace>...:
Failed to decode JSON: JSON message exceeded maximum buffer size...
```

The bridge auto-recovers (restarts the CLI subprocess) but the in-flight turn
is lost. Observed live 2026-07-20 11:22:07 (and previously 2026-07-18) during
image-heavy work.

Fix origin: OSS bridge repo (`coolcoolk/claude-code-telegram`), commit 01764ee.
Logic is identical between the OSS and template copies (only a pre-existing
surrounding-code offset at the opts dict site; the new lines are byte-identical).

No test breakage: no existing bridge test asserts the `opts` dict shape;
`bridge/tests/` grep clean for `opts[` / `max_buffer_size`.

Instances receive this fix on the next `update.sh` run against this release.
Live instances that already received a direct hotfix (Metal, Ag -- DGN-460
propagation-editor pass 2026-07-20) are already at the corrected state; this
release makes the fix canonical and consumable by all other version-consumers.
