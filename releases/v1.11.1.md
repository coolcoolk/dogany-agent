# Release notes -- v1.11.1 (2026-07-18)

Type: PATCH (2 fixes; no new surface, no behavior added beyond correctness
restoration).

## Summary

Two instance-affecting fixes: `update.sh` bridge-guard false-negative that
caused every instance to silently skip a bridge update (DGN-413), and
`relmod.py` global weekly hard-cap that silenced relationship-module alerts
for all people after any single alert fired in the prior 7 days (DGN-410).

## Fixed

### DGN-410 distribution gap: update.sh now syncs database/relmod.py

The v1.11.1 release carries a relmod.py behavior fix (see below), but
the fix was not included in the database file copy list on line 992 of
`update.sh`. Consequently, instances running `update.sh` after
consuming v1.11.1 would receive the updated framework but NOT the
updated relmod.py, leaving the weekly alert hard cap in effect on all
instances until a subsequent manual propagation or release.

Fix: added `relmod.py` to the database file copy list in `update.sh`.
Instances running `update.sh` after this fix will receive the complete
v1.11.1 delivery (both alert-cap removal and all other fixes).

---

### DGN-413: update.sh guard(ii) -- Vendor-rev marker now enforced

Background: DGN-385 established that a canonical bridge change must be
evidenced in `bridge/UPSTREAM.md` by either a `Pinned commit:` bump OR a
`Vendor-rev: <marker>` annotation. The annotation path is correct practice
when bridge content changes without a new upstream commit pin (bridge-content
vendor revision). However, `update.sh` guard(ii) (lines ~739-784) only
grepped for `Pinned commit:` and had no code path for `Vendor-rev:`.
`UPSTREAM.md` was also excluded from the rsync diff surface, so the marker
had zero enforcement power.

Consequence: DGN-399 (v1.11.0) correctly followed the discipline -- kept
the pin, added `Vendor-rev: DGN-399`. Every instance running `update.sh`
after consuming v1.11.0 saw "pins equal, N changed" and skipped the bridge
rsync entirely. Live incident: Kkari required manual 3-file recovery;
Metal's morning update was a silent no-op (though the code was present via
a prior local commit); Ag and Warg were at risk on their next update cycle.

Fix: guard(ii) decision table extended with 3 branches:
- (A) Pinned commit changed: proceed (existing behavior).
- (B) Vendor-rev present, pins equal: proceed (new branch).
- (C) Neither: skip (existing behavior).

Selftest T1-T5 all PASS, `bash -n` clean on canonical.

Propagation: canonical `dogany-agent/update.sh` patched (commit
8f2b6fa). All local instances receive this fix on their next `update.sh`
run against this release. Remote instances (kkari, shawn) receive it on
next manual update.

### DGN-410: relmod alert-pick -- CAP_REACHED removed, unexposed-first rotation added

Background: `database/relmod.py` `cmd_alert_pick` contained a global weekly
hard cap: if any relationship-module contact had been shown in the prior 7
days (`outcome='shown'`), the entire pick was aborted and `CAP_REACHED` was
printed. This silenced all alerts after the first week's picks, including
people who had never been shown and people who had been shown but not yet
acted on.

Live incident (2026-07-17/18): 3 people shown on 7/17; entire alert pick
muted on 7/18 -- including people in the unexposed pool.

Fix: the `CAP_REACHED` global cap block is removed entirely. In its place,
candidates are sorted by an unexposed-first rotation key:
`(last_shown is not None, last_shown or '', -ratio)`. This means:
1. People never shown (NULL last_shown) always sort first.
2. Among previously-shown people, oldest-shown sorts first.
3. Ratio is the tiebreak within each cohort.

The pool exhausts to previously-shown people only when the unexposed pool is
empty. Until a person is acted on (touch / snooze / dismiss / fade), they
remain in the candidate pool -- nobody is silently dropped between runs.

Unchanged: 3-pick limit per run; all per-person gates (let_fade, future
snooze, dismissed-cycle, upcoming-appointment exclusion, RESURFACE pass).

Selftest 29/29 green (26 original + 3 new: TC-9 updated; TC-9b rotation
continuity; TC-9c post-touch exclusion). Sole consumer of `CAP_REACHED`
output is `morning-brief.sh` (PICK line parser); cap removal causes no
regression there (confirmed by grep across all consumers).

Propagation: canonical `dogany-agent/database/relmod.py` patched (commit
677dca2). Ag consumes via `self-update` after this release ships. Template
does not carry `relmod.py` (relationship module is skill-bundle-delivered,
not rsync-channel). Spec source: Ag handoff YN379W5C verbatim.
